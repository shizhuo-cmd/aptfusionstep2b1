# Step2b1 攻击识别部分甲方讲解稿

## 1. 文档目的

这份文档面向甲方，用来完整说明 `APT-Fusionstep2b1` 版本中“攻击识别”这部分是怎么工作的、为什么这样设计、用了哪些技术、解决了哪些问题、还有哪些已知限制。

文档覆盖的范围是：

1. 从上游可疑任务图进入攻击分析开始。
2. 到最终输出攻击行为说明和 ATT&CK 战术/技术结果结束。
3. 重点讲 `module3`、`module4`、`module5`、`module6` 这四个阶段。

这份文档既讲“实现流程”，也讲“设计动机”和“技术取舍”。这样在向甲方讲解时，不会只停留在“代码做了什么”，还能讲清楚“为什么需要这样做”。

---

## 2. 一句话概括当前版本

当前 `step2b1` 的攻击识别，可以概括为：

1. 先由上游模型从海量图中筛出“可疑任务图”。
2. 再围绕这些任务图回查原始日志，恢复出与该任务相关的审计事件。
3. 然后把事件压缩成带状态的进程和对象，并按规则打标签。
4. 再利用父子进程关系和“同一对象先被写、后被另一进程读/执行”的因果关系，把离散事件拼成候选攻击链。
5. 最后把候选攻击链整理成结构化证据，结合规则、知识库和大模型，输出攻击行为解释和 ATT&CK 映射结果。

换句话说，这不是一个“直接把全量日志交给大模型”的系统，也不是一个“只靠单点异常分数输出结论”的系统，而是一个：

`图筛选 + 日志回查 + 规则标签 + 因果链构造 + 知识库检索 + 大模型解释`

的组合式攻击识别系统。

## 4. 整体架构和全流程

当前版本的攻击识别全流程分成四大阶段。

### 4.1 阶段总览

1. `module3_evidence`：围绕可疑任务图回查原始日志，恢复任务相关事件。
2. `module4_compact`：对恢复出的事件做压缩、状态汇总、轻量标签和标签来源记录。
3. `module5_paths`：做完整标签、上下文传播、跨对象因果连接、候选攻击链搜索和重排。
4. `module6_reason`：把候选攻击链解释成攻击行为，并映射到 ATT&CK。

### 4.2 全流程的输入和输出

#### 输入

1. 上游模块输出的可疑任务图列表。
2. 上游模块输出的任务结构信息。
3. 上游模块输出的重要进程/边贡献信息。
4. 原始审计日志。
5. 规则文件。
6. ATT&CK STIX 知识库。
7. 大模型推理接口。

#### 输出

1. 每个任务的标准化事件流。
2. 每个任务的进程状态、对象状态、对象版本和标签来源记录。
3. 每个任务的候选攻击链。
4. 每条攻击链的结构化证据包。
5. 每条攻击链对应的攻击行为说明。
6. 每条攻击链对应的 ATT&CK 战术/技术结果。

### 4.3 这套流程解决的核心问题

这套设计主要解决了七类问题：

1. 原始日志太多，不能直接全量解释。
2. 单靠上游图异常分数，无法给出可解释攻击过程。
3. 单靠父子进程关系，很多跨进程攻击动作串不起来。
4. 单靠对象共现，容易把无关事件错误串联。
5. 单靠规则标签，容易得到碎片化结果，缺乏完整攻击链。
6. 单靠大模型读原始日志，成本高、稳定性差、可复现性弱。
7. 单靠 ATT&CK 名称匹配，容易出现语义错误映射。

因此，这个版本采用的是“先结构化、再推理”的路线，而不是“直接黑箱推理”的路线。

## 5. 技术路线和设计动机

在详细讲模块之前，先讲清楚这版系统用了哪些技术，以及为什么要这样组合。

### 5.1 技术一：先用任务图做入口，而不是全量日志直接建事件图

#### 动机

全量日志直接建事件图虽然信息最全，但有几个问题：

1. 数据量太大，构图和搜索成本高。
2. 背景噪声极多，容易把正常系统行为也卷进来。
3. 大量无关对象和无关进程会稀释攻击信号。

#### 当前做法

先由上游模块识别“可疑任务图”，再以任务图中的进程为锚点回查日志。

#### 这样做的好处

1. 搜索范围可控。
2. 攻击分析聚焦在高风险区域。
3. 保留了图模型的召回能力，又避免了全量事件图过重的问题。

#### 取舍

1. 优点是效率高、噪声少。
2. 风险是：如果上游任务图本身漏掉了关键锚点，下游回查也可能漏证据。

### 5.2 技术二：先恢复事件，再做结构化压缩

#### 动机

原始日志字段不统一，不同数据集格式差异很大。

如果直接在原始日志上写规则，会遇到：

1. 同一个动作在不同数据集中名字不同。
2. 进程、文件、网络对象字段位置不同。
3. 时间格式不统一。
4. 同一进程可能存在多个编号表达方式。

#### 当前做法

先把原始日志统一为标准化事件，再在标准化事件上做后续所有逻辑。

#### 这样做的好处

1. 后面的标签、桥接、候选链搜索都基于统一事件模型。
2. 降低了对原始日志格式的耦合。
3. 支持跨数据集复用后续逻辑。

### 5.3 技术三：规则标签不是直接输出结果，而是作为中间语义层

#### 动机

直接从事件到 ATT&CK 映射会有两个问题：

1. 事件层太细碎，大模型难以稳定理解。
2. 单点事件语义不完整，容易误判战术阶段。

#### 当前做法

先把事件打成一层中间语义标签，再利用标签和因果关系构造候选攻击链。

#### 这样做的好处

1. 规则标签把底层系统调用提升为“外部接收”“可疑执行”“敏感读取”“横向连接”等更稳定的攻击语义。
2. 大模型面对的是结构化证据，而不是零散事件。

### 5.4 技术四：同时使用父子进程关系和跨对象因果关系

#### 动机

只靠父子进程关系会漏掉很多真实攻击动作：

1. 攻击者先让进程 A 写文件，再由进程 B 执行。
2. 攻击者先修改持久化对象，再由另一个系统进程读入。
3. 攻击者先打包归档，再由另一个进程外传。

这些都不是单纯的父子进程链。

#### 当前做法

系统同时使用两类边：

1. 父子进程边。
2. 同一对象先写后读/执行的跨对象因果边。

#### 这样做的好处

1. 可以把“对象传播”纳入攻击链。
2. 攻击链不再只是一条进程树路径，而是带对象传播语义的因果链。

### 5.5 技术五：语义去重和事件聚合

#### 动机

审计日志中存在大量重复操作，例如：

1. 高频读取同一文件。
2. 反复访问同一网络流。
3. 大量重复的库加载或近似事件。

如果不压缩：

1. 候选链会被噪声淹没。
2. 大模型输入过长。
3. 同一个攻击行为会被无意义重复。

#### 当前做法

用语义键、标签签名和纪元机制做事件去重，再按时间桶聚合事件片段。

#### 这样做的好处

1. 降低噪声。
2. 保留关键状态变化。
3. 让后续链条搜索更稳定。

### 5.6 技术六：规则 + 知识库检索 + 大模型三段式输出

#### 动机

如果只靠规则：

1. 可解释性强，但表达能力有限。
2. 很难稳定映射到更高层战术语义。

如果只靠大模型：

1. 泛化能力强，但容易幻觉。
2. 难以复现和验证。

如果只靠 ATT&CK 名称匹配：

1. 很容易词面相似但语义不对。

#### 当前做法

1. 先用规则抽象出 Holmes 风格行为原子。
2. 再用本地知识库检索 ATT&CK 候选。
3. 最后让大模型在“有证据、有候选、有约束”的条件下做映射。

#### 这样做的好处

1. 规则层保证底线正确性。
2. 知识库层限制搜索空间。
3. 大模型层提供最终语义整合能力。

## 6. 阶段一：围绕可疑任务图回查原始日志

这一阶段对应 `module3_evidence`。

### 6.1 这一步为什么必要

上游任务图只告诉系统“这里可能有问题”，但并不包含完整攻击证据。

要真正解释攻击，必须回到原始日志，把：

1. 具体系统调用
2. 访问对象
3. 网络目标
4. 时间顺序
5. 父子进程关系

全部恢复出来。

因此 `module3` 的目标不是判定攻击，而是把“任务图锚点”还原成“事件级证据流”。

### 6.2 这一步用了哪些技术

1. 日志解析技术
2. 数据集适配技术
3. 进程别名归一
4. 局部前沿扩展
5. 标准化事件建模
6. 对象类型分类

### 6.3 输入文件

主要依赖上游输出：

1. `module2/suspicious_tasks.json`
2. `module2/task_meta_rich.json`
3. `module2/task_attribution.json`

### 6.4 任务筛选规则

当前版本支持三种任务筛选方式：

1. `predicted_positive`
   - 只取上游预测为可疑的任务
2. `ground_truth_positive`
   - 只取真实标注为恶意的任务
3. `ground_truth_positive_base_only`
   - 只取真实恶意且不是增强任务的基础任务

当前默认采用：

1. `predicted_positive`

### 6.5 任务严重等级规则

系统会根据任务分数在全体任务中的百分位给出严重等级：

1. 百分位大于等于 `99`：`Critical`
2. 百分位大于等于 `97`：`Significant`
3. 百分位大于等于 `95`：`Moderate`
4. 百分位大于等于 `90`：`Minor`
5. 其他：`Negligible`

随后系统又做了一个保守修正：

1. `Minor` 被提升为 `Moderate`
2. `Negligible` 也被提升为 `Moderate`

这样做的目的，是避免过低的分级在后续攻击分析里被过早弱化。

### 6.6 种子来源规则

日志回查的种子来自任务图里的进程编号集合。

规则是：

1. 把任务图中的所有进程编号作为初始前沿集合。
2. 扫描原始日志。
3. 如果一条日志的主体进程命中前沿，就把这条日志归入该任务。
4. 如果开启“对象侧匹配”，则事件对象命中前沿也会归入该任务。

当前默认开启对象侧匹配。

#### 这样设计的动机

1. 只用主体匹配，可能漏掉“对象侧被引用”的关键事件。
2. 加入对象侧匹配，可以补回一部分通过对象传播的证据。

### 6.7 局部跳数扩展规则

系统支持局部扩展跳数：

1. `1` 跳：只保留直接命中种子的事件。
2. `2` 跳：如果第一轮命中的事件里对象本身也是进程，则把该进程加入下一轮前沿，再继续扫一轮。

当前版本只允许 `1` 或 `2`，默认是 `1`。

#### 动机

1. 如果完全不扩展，可能漏掉短距离关联进程。
2. 如果无限扩展，任务范围会迅速膨胀，噪声过高。

所以这里只允许非常保守的局部扩展。

### 6.8 不同数据集的日志解析规则

当前实现适配了三类来源：

1. DARPA TC3 / NODLINK 风格日志
2. OPTC 风格日志
3. NODLINK Windows 原始日志

不管输入来自哪类日志，最后都会被统一成“标准化事件”。

### 6.9 进程别名归一规则

这是一个非常关键但经常被忽略的问题。

#### 为什么需要做

同一个真实进程，在原始日志里可能有不同编号表示方式：

1. 原始 UUID
2. 系统内部进程 ID
3. 规范化后的进程 ID

如果不先归一：

1. 父子关系会断裂
2. 进程链会被拆散
3. 对象访问会错配到不同进程

#### 当前做法

1. `trace` 主机
   - 以 `cid` 作为规范进程编号
   - 原始 UUID 映射到 `cid`
2. `theia`
   - 用 `(父进程, tgid, path)` 组合做归并
3. `optc` 与 Windows 原始日志
   - 主要使用原始进程编号

### 6.10 事件标准化规则

日志事件会被统一映射到标准事件类型。

当前显式映射包括：

1. `EXECUTE -> EXEC`
2. `CREATE_OBJECT -> CREATE`
3. `LOAD_ELF -> LOAD`
4. `LOADLIBRARY -> LOAD`
5. `SENDMSG -> SEND`
6. `SENDTO -> SEND`
7. `RECVMSG -> RECV`
8. `RECVFROM -> RECV`
9. `FORK_WITH_SHARED_OPEN_FILE -> FORK`
10. `OPEN_WITH_CLOSE_ON_EXEC -> OPEN`
11. `MODIFY_PROCESS -> CHMOD`

#### 动机

只有先做动作归一，后面的规则才能跨数据集复用。

### 6.11 进程信息提取规则

#### 可执行路径提取

1. 从进程属性字符串中取第一个像路径的 token。

#### 进程名提取

1. 优先从可执行路径 basename 提取。
2. 如果没有路径，再从原始进程属性的第一个 token 提取 basename。

#### 命令行提取

1. 如果进程属性中存在空格，则保留整串作为命令行。

### 6.12 对象键和对象名提取规则

1. 如果对象是进程，直接使用该进程编号。
2. 如果对象不是进程，优先使用对象属性作为对象键。
3. 如果对象属性为空，则退回使用对象 UUID。
4. 对象名通常取 basename；网络对象则保留完整四元组。

### 6.13 网络端点解析规则

如果对象键中出现：

`源IP:源端口->目标IP:目标端口`

系统就解析出：

1. 本地 IP / 端口
2. 远端 IP / 端口

如果只出现单个 `IP:端口`，则把它视作远端端点。

### 6.14 系统调用方向与语义流向规则

#### 系统调用方向

1. `FORK / CLONE / EXIT`：进程到进程
2. `ACCEPT` 且对象是网络：对象到进程
3. 对象本身是进程：进程到进程
4. 其他有对象的情况：进程到对象

#### 语义流向

1. `READ / EXEC / LOAD / MMAP`：对象流向进程
2. `WRITE / CREATE / TRUNCATE / CHMOD / CHOWN / RENAME / DELETE / CONNECT / SEND`：进程流向对象
3. `RECV / ACCEPT`：对象流向进程
4. `FORK / CLONE`：进程流向进程
5. `EXIT`：无语义流向

#### 动机

这是后面构造因果链和对象传播关系的基础。

### 6.15 对象分类规则

对象分类是后续标签体系的基础。

#### 网络类对象

1. 远端或本地 IP 只要命中外部地址，就归为外部 IP
2. 否则命中内网网段，就归为内部 IP
3. `127.*`、`localhost`、`unix:` 归为本地通信对象

#### 路径类对象

按顺序匹配：

1. `/proc/*`：进程状态对象
2. `/tmp/`、`/var/tmp/`、`/dev/shm/`：临时文件
3. 凭据路径：如 `/etc/passwd`、`/etc/shadow`、`.ssh/id_rsa`、`.aws/credentials`
4. 历史记录路径：如 `.bash_history`、`.zsh_history`
5. 持久化路径：如 `/etc/crontab`、`/etc/cron.d`、`/etc/systemd/system`
6. 提权配置路径：如 `/etc/sudoers`
7. 认证配置路径：如 `/etc/ssh/`、`/etc/pam.d/`
8. 日志路径：如 `/var/log`、`/var/log/auth.log`
9. 压缩归档后缀：`.zip`、`.tar`、`.gz`、`.7z`
10. 业务数据路径：如 `*.db`、`*.sqlite`、`*.xlsx`、`*.csv`
11. 系统库路径：如 `/usr/lib`、`/lib`
12. 系统资源路径：如 `/usr/share`

#### 动机

这一步把低层对象语义提升到安全分析可用的层级，为后续标签规则服务。

### 6.16 去重规则

一条事件如果与同任务中的另一条事件在以下六元组完全相同，就不重复计入：

1. 主体进程编号
2. 对象编号
3. 动作
4. 时间戳
5. 对象类型提示
6. 对象属性提示

#### 动机

避免同一条日志被多次扫描命中时重复加入任务。

### 6.17 每任务最大事件上限

每个任务最多保留：

1. `300000` 条事件

超出的事件会被记录为“超过上限丢弃”。

#### 动机

这是一个工程保护措施，用于防止个别超大任务拖垮整个管线。

### 6.18 本阶段输出

这一阶段输出的核心产物包括：

1. 标准化事件流
2. 实体索引
3. 进程到事件索引
4. 对象到事件索引
5. 任务局部证据前沿
6. 任务局部证据图

#### 说明

这些 sidecar 文件对人工分析很有价值，但后续主流程真正大量使用的，主要还是标准化事件流和任务先验信息。

## 7. 阶段二：事件压缩、状态构造、轻量标签和标签来源

这一阶段对应 `module4_compact`。

### 7.1 这一步为什么必要

从 `module3` 出来的仍然是事件流，而不是适合攻击链搜索的结构化状态。

直接在事件流上做攻击链搜索，会遇到：

1. 重复事件太多
2. 进程和对象状态难以维护
3. 文件先写后执行、先写后读取等关系不好追踪
4. 标签难以积累和继承

因此，这一步的目标是把“事件流”变成“状态图”。

### 7.2 当前用了哪些技术

1. 事件级状态压缩
2. 进程状态汇总
3. 对象状态汇总
4. 对象版本追踪
5. 语义去重
6. 轻量标签预标记
7. 标签来源记录
8. 事件聚合

### 7.3 进程继承规则

当事件类型为 `FORK` 或 `CLONE`，且对象本身是进程时：

1. 子进程继承父进程当前全部上下文标签
2. 如果子进程状态尚不存在，则创建子进程状态
3. 如果子进程状态已存在，则补齐尚未拥有的上下文标签
4. 这一步会写标签来源记录

#### 动机

攻击上下文通常会沿进程衍生链传递，例如：

1. Web 服务进程衍生 Shell
2. Shell 再衍生解释器
3. 解释器再衍生下载器

如果不在进程分叉时做基础上下文继承，后续很多链会断掉。

### 7.4 轻量标签规则

这一阶段的标签规则属于“低成本预标记”，目的是先把非常确定的上下文和对象语义标出来。

下面逐条列出当前真实执行的规则。

#### 7.4.1 Web 服务上下文

中文含义：

1. 这个进程处在 Web 服务相关上下文中

触发规则：

1. 进程名属于 Web 服务名单：
   - `nginx`
   - `httpd`
   - `apache2`
   - `php-fpm`
   - `php-cgi`
   - `tomcat`
   - `uwsgi`
   - `gunicorn`
   - `node`
2. 或者对带上传标记的路径执行 `WRITE / CREATE`，且该路径属于 Web 根目录

#### 7.4.2 远程服务上下文

中文含义：

1. 这个进程处在远程登录/远程服务上下文中

触发规则：

1. 进程名属于远程服务名单：
   - `sshd`
   - `dropbear`
   - `telnetd`
   - `xrdp`
   - `sftp-server`

#### 7.4.3 外部网络对象

中文含义：

1. 当前对象是对外通信对象

触发规则：

1. 对象类别是外部 IP

#### 7.4.4 外部网络接触上下文

中文含义：

1. 当前进程和外部网络发生过接触

触发规则：

1. 对象类别是外部 IP
2. 事件属于 `CONNECT / ACCEPT / SEND / RECV`

#### 7.4.5 对象类别直标规则

系统会把对象类别直接映射成对象标签：

1. 临时文件 -> 临时文件对象标签
2. 凭据文件 -> 凭据对象标签
3. 历史文件 -> 历史对象标签
4. 业务文件 -> 业务数据对象标签
5. 持久化文件 -> 持久化对象标签
6. 提权配置 -> 提权配置对象标签

#### 7.4.6 上传文件规则

中文含义：

1. 当前对象看起来像被上传到 Web 目录或上传目录的文件

触发规则：

1. 事件类型是 `WRITE / CREATE`
2. 对象路径中包含：
   - `upload`
   - `uploads`
   - `wwwroot`
   - `webapps`
   - `htdocs`

附加效果：

1. 如果该路径位于 Web 根目录，进程还会同时获得 Web 服务上下文标签

#### 7.4.7 不存在文件规则

中文含义：

1. 进程访问了一个原本应在临时区域中的对象，但该对象不存在

触发规则：

1. 对象类别是临时文件
2. 事件类型属于 `READ / OPEN / EXEC / LOAD / MMAP`
3. 返回结果是 `ENOENT` 或 `NOT_FOUND`

#### 7.4.8 下载文件规则

中文含义：

1. 当前对象看起来像从外部下载或接收得到的文件

触发规则满足其一即可：

1. 命令行里有下载提示：
   - `http://`
   - `https://`
   - `ftp://`
   - `-o`
   - `-O`
   且事件类型为 `WRITE / CREATE`
2. 进程名属于下载器名单，且在 120 秒内出现过外联或外部接收，再发生 `WRITE / CREATE`
3. 对象类别是临时文件，且同一进程在 120 秒内出现过外部接收，再发生 `WRITE / CREATE`

#### 7.4.9 不可信上下文规则

中文含义：

1. 当前进程已经接触过外部输入，或者已经执行了可疑来源对象

触发规则满足其一即可：

1. 发生 `RECV`，对象是外部 IP
2. 发生 `EXEC / LOAD / MMAP`，且对象已经具有：
   - 下载文件标签
   - 上传文件标签
   - 疑似落地后执行对象标签

### 7.5 标签来源记录规则

当前版本会为以下两类标签生成来源记录：

1. 轻量标签预标记产生的标签
2. `FORK / CLONE` 继承产生的标签

每条标签来源记录都会保存：

1. 标签名
2. 标签类型
3. 持有者是谁
4. 来源实体是谁
5. 来源事件是谁
6. 是哪条规则生成的
7. 上游标签来源编号

#### 动机

这样做的目的，是保留“标签从哪里来”的可追溯性。

#### 当前问题

这套来源记录目前还不是全量覆盖：

1. `module5` 里后续补上的很多行为标签，并不会继续写入这份来源记录
2. 因而它目前更像“局部标签履历”，还不是“完整标签因果链”

### 7.6 对象版本规则

系统把同一个对象视为可能经历多个版本。

规则如下：

1. 对象第一次出现时，创建 `v0001`
2. 如果事件类型属于：
   - `WRITE`
   - `CREATE`
   - `TRUNCATE`
   - `RENAME`
   - `DELETE`
   - `CHMOD`
   - `CHOWN`
   则推进到新版本
3. 每个版本上累计记录：
   - 版本起止时间
   - 该版本标签
   - 写者进程集合
   - 读者进程集合
   - 执行者进程集合

#### 动机

如果不做版本划分，同一个文件在不同时间被不同进程读写，会被错误看成同一份内容，容易造成错误桥接。

### 7.7 语义去重规则

这是当前版本降低噪声最关键的技术之一。

#### 为什么需要语义去重

审计日志中存在大量重复但没有新增信息的事件：

1. 反复读取同一文件
2. 高频访问同一对象
3. 连续相同的近似事件

如果每条都保留：

1. 事件量会爆炸
2. 大模型输入会过长
3. 候选攻击链会被重复事件淹没

#### 语义键定义

系统为每个事件构造一个语义键，由下面六项组成：

1. 任务编号
2. 进程编号
3. 事件类型
4. 对象键
5. 对象类别
6. 语义流向

#### 只有同时满足下面条件，事件才会被认定为可去重

1. 语义键相同
2. 进程标签签名相同
3. 对象标签签名相同
4. 对象语义纪元相同
5. 进程控制纪元相同
6. 与上一次出现的时间间隔不超过 `600` 秒

#### 什么情况下绝不去重，必须保留

满足其一就强制保留：

1. `EXEC`
2. 对外网络事件：`CONNECT / ACCEPT / SEND / RECV`
3. 对敏感对象的写改删权限操作
4. 事件类型属于强制保留集合：
   - `EXEC`
   - `FORK`
   - `CLONE`
   - `CONNECT`
   - `ACCEPT`
   - `SEND`
   - `RECV`
   - `EXIT`
   - `CHMOD`
   - `CHOWN`
   - `RENAME`
   - `DELETE`
5. 对象类别属于强制保留类别：
   - 临时文件
   - 凭据文件
   - 历史文件
   - 业务文件
   - 持久化文件
   - 提权配置
   - 外部 IP
6. 当前事件触发了任意标签

#### 纪元失效规则

以下变化会让旧语义记忆失效：

1. 对象被写改删时，对象语义纪元加一
2. 进程收到外部数据时，进程控制纪元加一
3. 进程执行/加载/映射对象时，进程控制纪元加一

#### 设计动机

这里不是简单“按事件文本去重”，而是“按语义状态去重”。

这能解决一个常见问题：

1. 两条事件表面一样，但如果对象内容已经变了，或者进程上下文已经变了，它们就不应被当成同一事件。

### 7.8 重要对象规则

以下对象会被记录为进程的重要对象：

1. 当前事件触发了任意标签的对象
2. 对象类别属于：
   - 凭据文件
   - 历史文件
   - 业务文件
   - 持久化文件

#### 动机

后续构造候选攻击链时，需要知道哪些对象真正有安全语义价值，而不是把所有普通对象一股脑带进去。

### 7.9 允许桥接的对象规则

如果对象带有以下对象标签之一，就会被视为允许参与跨进程因果桥接：

1. 临时文件对象标签
2. 下载文件对象标签
3. 上传文件对象标签
4. 疑似落地后执行对象标签
5. 归档对象标签
6. 持久化对象标签
7. 提权配置对象标签

### 7.10 事件聚合规则

系统还会把保留事件聚合成时间桶内的“事件片段”。

聚合键包括：

1. 时间桶
2. 进程编号
3. 事件类型
4. 对象类型
5. 对象类别
6. 对象键
7. 语义流向
8. 进程标签签名
9. 对象标签签名
10. 对象语义纪元
11. 进程控制纪元

默认：

1. 时间桶大小 `1` 分钟
2. 每个片段最多保留 `5` 个代表事件

#### 动机

这是为后续展示和人工复盘服务，目的是让同类事件更易读。

### 7.11 本阶段输出

1. 保留事件
2. 对象访问记录
3. 事件片段
4. 进程状态
5. 对象状态
6. 对象版本
7. 标签来源记录

## 8. 阶段三：完整标签、上下文传播、跨对象因果链和候选攻击链

这一阶段对应 `module5_paths`，是整个攻击识别流程的核心。

### 8.1 为什么这一阶段是核心

前两步只是在“准备证据”和“做状态压缩”。

真正决定系统能不能识别出攻击过程的，是这一阶段的四件事：

1. 是否能把关键攻击行为打成标签
2. 是否能把上下文沿进程树传播下去
3. 是否能把“同一对象传播”转成进程间因果关系
4. 是否能把这些关系拼成合理的候选攻击链

### 8.2 当前用了哪些技术

1. 规则标签增强
2. 上下文传播
3. 跨对象因果边构造
4. 深度优先候选链搜索
5. 风险打分
6. 证据质量重排
7. 模式多样性保留
8. 候选链摘要生成

### 8.3 完整对象标签规则

#### 8.3.1 疑似落地后执行对象

中文含义：

1. 同一个对象先被写入或创建，随后又被执行/加载

触发规则：

1. 对象不能属于系统库或系统资源
2. 同一对象的访问序列里先出现：
   - `WRITE`
   - `CREATE`
   - `RENAME`
3. 后面再出现：
   - `EXEC`
   - `LOAD`
   - `MMAP`

#### 动机

这是识别“落地载荷再执行”这一典型攻击模式的关键规则。

### 8.4 完整行为标签规则

下面所有规则都是真正在当前代码里执行的完整行为标签规则。

#### 8.4.1 外部接收

触发条件：

1. 事件类型是 `RECV`
2. 对象类别是外部 IP

#### 8.4.2 外部发送

触发条件：

1. 事件类型是 `SEND`
2. 对象类别是外部 IP

#### 8.4.3 临时目录执行

触发条件：

1. 事件类型是 `EXEC`
2. 对象类别是临时文件

#### 8.4.4 执行下载文件

触发条件：

1. 事件类型是 `EXEC / LOAD`
2. 对象已被标记为下载文件

#### 8.4.5 执行上传文件

触发条件：

1. 事件类型是 `EXEC / LOAD`
2. 对象已被标记为上传文件

#### 8.4.6 执行疑似落地对象

触发条件：

1. 事件类型是 `EXEC / LOAD`
2. 对象已被标记为疑似落地后执行对象

#### 8.4.7 可疑 Shell 执行

前置条件：

1. 进程名属于 Shell：
   - `sh`
   - `bash`
   - `zsh`
   - `dash`

再满足以下任意一条：

1. 进程当前已有：
   - Web 服务上下文
   - 远程服务上下文
   - 可疑控制上下文
2. 事件中父进程名属于 Web 服务
3. 命令行中包含：
   - ` -c `
   - `curl`
   - `wget`
   - `nc `
   - `socat`
   - `/tmp/`
   - `/dev/shm/`

#### 动机

Shell 本身并不一定恶意，因此这里只把“处于可疑上下文、或命令行明显具有攻击痕迹”的 Shell 作为攻击行为。

#### 8.4.8 可疑解释器执行

前置条件：

1. 进程名属于解释器：
   - `python`
   - `python3`
   - `perl`
   - `php`
   - `node`
   - `ruby`
   - `lua`

再满足任意一条：

1. 命令行包含：
   - `http://`
   - `https://`
   - `/tmp/`
   - `/dev/shm/`
   - `-c `
2. 进程当前具有：
   - Web 服务上下文
   - 远程服务上下文
   - 可疑控制上下文
3. 当前事件带有以下对象触发信息之一：
   - 下载文件
   - 上传文件
   - 疑似落地后执行对象

#### 8.4.9 解释器启动

触发条件：

1. 进程名属于解释器即可

#### 动机

这个标签强度较低，主要用于补充阶段覆盖，不单独代表恶意结论。

#### 8.4.10 读取凭据

触发条件：

1. 事件类型是 `READ`
2. 对象类别是凭据文件

附带效果：

1. 进程追加“高价值访问上下文”

#### 8.4.11 读取历史记录

触发条件：

1. 事件类型是 `READ`
2. 对象类别是历史文件

附带效果：

1. 进程追加“高价值访问上下文”

#### 8.4.12 读取业务数据

触发条件：

1. 事件类型是 `READ`
2. 对象类别是业务文件

附带效果：

1. 进程追加“高价值访问上下文”

#### 8.4.13 大量文件访问

当前真实执行规则：

1. 以“进程 + 分钟桶”为窗口
2. 统计该窗口内被该进程读取过的不同对象数
3. 若对象数达到 `100`，则打上“大量文件访问”标签

#### 设计说明

规则文件里写的是“5 分钟窗口 + 100 个文件或 100MB 读取量”，但当前代码只实现了“1 分钟内 100 个不同对象”。

这是当前版本实现和设计声明不完全一致的一个典型点。

#### 8.4.14 写持久化位置

触发条件：

1. 事件类型是 `WRITE / CREATE / RENAME`
2. 对象类别是持久化文件

#### 8.4.15 修改提权配置

触发条件：

1. 事件类型是 `WRITE / CHMOD / CHOWN`
2. 对象类别是提权配置

#### 8.4.16 压缩归档

满足任意一条：

1. 进程名属于归档工具：
   - `tar`
   - `zip`
   - `gzip`
   - `bzip2`
   - `xz`
   - `7z`
2. 对象类别是归档文件

#### 8.4.17 清理日志

触发条件：

1. 事件类型属于：
   - `DELETE`
   - `RENAME`
   - `WRITE`
2. 对象类别是日志文件

#### 8.4.18 横向连接

触发条件：

1. 事件类型是 `CONNECT / SEND`
2. 对象类别是内部 IP
3. 远端端口属于横向高风险端口：
   - `22`
   - `445`
   - `3389`
   - `3306`
   - `5432`
   - `6379`
   - `9200`
   - `27017`

#### 8.4.19 远程服务入口活动

触发条件：

1. 进程名属于远程服务
2. 事件类型是 `ACCEPT / RECV`

#### 8.4.20 Web 上传写入

触发条件：

1. 事件类型是 `WRITE / CREATE`
2. 对象键中包含 `upload`

### 8.5 上下文传播规则

当前传播的不是所有标签，而是“上下文标签”。

#### 允许传播的标签

1. Web 服务上下文
2. 远程服务上下文
3. 外部网络接触上下文
4. 不可信上下文
5. 高价值访问上下文
6. 可疑控制上下文

#### 各标签最大传播深度

1. 高价值访问上下文：`1`
2. 外部网络接触上下文：`2`
3. Web 服务上下文：`2`
4. 远程服务上下文：`4`
5. 不可信上下文：`3`
6. 可疑控制上下文：`3`

#### 公共守护进程传播限制

如果父进程属于公共守护进程：

1. `systemd`
2. `cron`
3. `crond`
4. `sshd`
5. `nginx`
6. `apache2`
7. `httpd`
8. `mysqld`
9. `redis-server`
10. `rsyslogd`
11. `dbus-daemon`

那么只有当子进程类型属于下列集合时，才允许传播：

1. shell
2. 解释器
3. 下载器
4. 网络工具
5. 未知二进制

#### 聚合规则：子进程可疑

如果某个父进程的任一子进程满足其一：

1. 子进程已有行为标签
2. 子进程具有不可信上下文
3. 子进程具有可疑控制上下文

则父进程获得“子进程可疑”聚合标签。

#### 设计动机

攻击链往往沿子进程向下展开，但系统不能无限制传播，否则正常系统服务也会被污染。

因此这里采用了：

1. 只传播上下文，不传播行为
2. 控制最大层数
3. 限制公共守护进程向普通子进程传播

### 8.6 跨对象因果边规则

这是当前版本最有特点的机制之一。

#### 中文解释

如果同一个对象先被进程 A 写入或修改，之后又被进程 B 读取、执行或加载，那么系统会认为：

1. 进程 A 和进程 B 之间可能存在一条通过该对象传递的因果关系

这条关系就是“跨对象因果边”。

#### 为什么需要它

很多真实攻击链都不是简单的父子进程链，而是：

1. A 下载文件，B 执行文件
2. A 修改持久化文件，B 读取后启动
3. A 产出归档文件，B 再外传

如果没有这类因果边，系统会漏掉大量攻击传播链条。

#### 允许参与桥接的对象条件

对象必须同时满足：

1. 对象类别不能属于：
   - 系统库
   - 系统资源
   - `/proc` 状态对象
2. 对象标签不能命中禁止集合：
   - 凭据对象
   - 历史对象
   - 业务数据对象
   - 外部网络对象
   - 内部网络对象
   - 认证配置对象
   - 安全日志对象
3. 对象标签必须命中允许集合：
   - 临时文件对象
   - 下载文件对象
   - 上传文件对象
   - 疑似落地后执行对象
   - 归档对象
   - 持久化对象
   - 提权配置对象

#### 写端事件类型

允许作为因果边写端的事件：

1. `WRITE`
2. `CREATE`
3. `RENAME`

#### 读/执行端事件类型

允许作为因果边后续端的事件：

1. `READ`
2. `EXEC`
3. `MMAP`
4. `LOAD`

#### 额外约束

还必须同时满足：

1. 写者进程和后续进程不能是同一个
2. 后续时间不能早于写入时间
3. 时间差不能超过 `30` 分钟
4. 写后对象语义纪元必须等于后续访问前对象语义纪元
5. 每个对象最多生成 `20` 条桥接边

#### 因果边类型

1. 后续事件是 `EXEC / LOAD / MMAP`：写后执行
2. 对象属于持久化对象：持久化后续使用
3. 对象属于归档对象：归档后续使用
4. 其他：写后读取

#### 因果边置信度

1. 执行的是临时/下载/上传/疑似落地对象：`0.93`
2. 持久化或提权配置对象：`0.78`
3. 归档对象：`0.66`
4. 其他：`0.55`

#### 附加聚合标签

如果桥接对象属于：

1. 下载文件
2. 上传文件
3. 疑似落地后执行对象

则桥边两端进程都追加“通过可疑对象桥接”聚合标签。

### 8.7 候选攻击链搜索规则

系统在一个混合图上搜索候选攻击链。

#### 搜索图由什么组成

1. 父子进程边
2. 跨对象因果边

#### 起始种子规则

只要进程满足任意一条，就能作为候选链起点：

1. 有任意行为标签
2. 有 Web 服务上下文
3. 有远程服务上下文
4. 有不可信上下文
5. 有可疑控制上下文

#### 深度约束

1. 候选链最大进程长度：`6`

#### 时间约束

1. 整条链最大跨度：`180` 分钟
2. 相邻扩展最大时间间隔：`120` 分钟
3. 时间必须单调向前

#### 环路约束

1. 候选链内不能重复出现同一进程

#### 阶段覆盖规则

每个标签都被映射到一个攻击阶段：

1. `Entry`：入口或初始接触
2. `ExecutionWeak`：弱执行迹象
3. `ExecutionStrong`：强执行迹象
4. `TargetAccess`：目标访问/敏感读取
5. `FollowUp`：后续动作，如持久化、横向、外传、清理

候选链只有在阶段集合满足下面任意一种时才会被保留。

##### 强阶段组合

1. `Entry + ExecutionStrong + TargetAccess`
2. `Entry + ExecutionStrong + FollowUp`
3. `ExecutionStrong + TargetAccess + FollowUp`

##### 中等升级组合

1. `Entry + ExecutionWeak + TargetAccess`
2. `Entry + ExecutionWeak + FollowUp`

##### 弱阶段组合

1. `Entry + ExecutionWeak`
2. `Entry + TargetAccess`
3. `ExecutionWeak + FollowUp`

#### 设计动机

系统不要求每条候选链都必须完整覆盖所有阶段，否则召回会太低。

因此设计成：

1. 强组合优先
2. 中等组合次之
3. 弱组合作为保底召回

### 8.8 候选攻击链去重规则

当前去重键只看“进程链本身”。

这意味着：

1. 如果两条链进程序列完全相同，只是来自不同起点重复搜索，或桥接边来源不同，则视为同一条
2. 如果两条链只是前半段相同，后半段不同，则都会保留
3. 如果一条链是 `A-B`，另一条是 `A-B-C`，因为进程序列不同，也都会保留
4. 如果同一条进程序列被多次搜索到，优先保留阶段覆盖更丰富的那条

### 8.9 候选攻击链基础打分规则

总分公式是：

`标签分 + 组合分 + 阶段分 + 桥接分 + 上游先验分 - 惩罚分`

#### 标签分

主要高价值标签分数如下：

1. 不可信上下文：`5`
2. 高价值访问上下文：`8`
3. 可疑控制上下文：`8`
4. 外部接收：`8`
5. 外部发送：`20`
6. 临时目录执行：`20`
7. 执行下载文件：`30`
8. 执行疑似落地对象：`28`
9. 读取凭据：`15`
10. 读取业务数据：`18`
11. 写持久化位置：`30`
12. 修改提权配置：`30`
13. 清理日志：`25`
14. 横向连接：`20`

#### 阶段分

1. 每覆盖一个阶段，加 `12`

#### 桥接分

1. 每条桥接边加 `桥接置信度 * 12`

#### 组合加分

当前真实执行的组合规则包括：

1. 外部接收 + 临时目录执行 或 外部接收 + 执行下载文件
2. 外部接收 + Shell
3. 执行下载/疑似落地对象 + 读取凭据
4. 读取凭据/业务数据 + 外部发送
5. 执行下载/疑似落地对象 + 持久化写入
6. 执行下载/疑似落地对象 + 横向连接
7. 链长至少 3 且阶段数至少 3

#### 上游先验分

包含三部分：

1. 任务异常分数乘权重
2. 若链中包含上游模型认为重要的进程，则按排名加分
3. 若链中包含上游模型认为重要的边，则按排名加分

时间一致性修正规则：

1. 若候选链时间完全落在任务时间范围内，加 `8`
2. 否则减 `12`

#### 惩罚项

当前真实执行的惩罚项有：

1. 只有弱执行、没有强执行
2. 桥接边数量达到 `4` 条及以上
3. 只有一个阶段且只是敏感读取

### 8.10 支撑证据整理规则

系统不会只给出一条“进程序列”，还会整理出用于解释该链的辅助证据。

#### 支撑证据事件

来源包括：

1. 链上进程自身的证据事件
2. 所有桥接边对应的写事件和读/执行事件
3. 链上触发标签的保留事件

#### 支撑证据对象

来源包括：

1. 链上进程的重要对象
2. 桥接边涉及对象
3. 支撑事件涉及对象

#### 支撑证据关系

来源包括：

1. “进程 A 经对象 X 影响进程 B”的桥接关系摘要
2. 对象版本的写者/读者/执行者摘要

#### 前导事件规则

若事件文本命中以下关键词，则记为前导事件：

1. `tcexec`
2. `command-not-found`
3. `/dev/pts/3`
4. `python3`
5. `chmod`
6. `bash`

#### 后续动作事件规则

满足任意一条则记为后续动作事件：

1. 事件带：
   - 外部发送
   - 横向连接
   - 清理日志
2. 对象类别为外部或内部 IP，且事件属于网络收发
3. 删除了临时目录中的对象

#### 网络摘要

会统计：

1. 外部接收次数
2. 外部发送次数
3. 内部连接次数
4. 唯一外部目标数
5. 唯一内部目标数

#### 对象脉络摘要

优先输出：

1. 可疑桥对象如何在不同进程间传播

若没有明确桥对象，则退化为输出：

1. 若干重要对象版本的写者/读者/执行者数量

### 8.11 候选链补救规则

系统额外有一个“前导分支补救”机制。

触发条件：

1. 当前任务中还没有“短生命周期解释器前导分支”类型候选链
2. 支撑事件中至少有 2 条命中前导关键词

然后系统会：

1. 收集前导事件涉及的进程
2. 适度补入共享父进程且时间邻近的进程
3. 生成一条最多 6 个进程的补救链

#### 动机

某些攻击在最前段只留下很短的解释器或脚本落地动作，如果完全按常规搜索，很容易被后续更强路径淹没。

### 8.12 模式多样性保留规则

最终保留前 `path_top_k` 条路径时，不是简单按分数截断，而是会先尝试保住不同行为模式各自的一条代表路径。

优先保留的模式包括：

1. 短前导解释器分支
2. 附件/用户触发执行
3. 临时目录或落地文件执行
4. 外部回连
5. 内部扫描或横向
6. 痕迹清理
7. 邮件/浏览器相关上下文尾部

#### 动机

这样做是为了避免前几名候选链全都属于同一种模式，导致分析报告缺少覆盖面。

### 8.13 证据质量重排规则

在基础打分之后，系统还会做一次小幅度重排。

注意：

1. 这一步不是重新构链
2. 而是对已有候选链做“证据质量修正”

修正幅度限制在：

1. `-4 ~ +4`

#### 证据紧凑度

根据支撑事件的总时间跨度调整：

1. `<= 5` 分钟：`+2`
2. `<= 15` 分钟：`+1`
3. `<= 45` 分钟：`0`
4. `<= 90` 分钟：`-1`
5. `<= 180` 分钟：`-2`
6. `> 180` 分钟：`-3`

#### 标签来源覆盖度

1. 先取链上的关键标签
2. 再看这些标签里有多少能在标签来源记录中找到对应支撑

打分规则：

1. 覆盖率 `>= 0.85`：`+3`
2. 覆盖率 `>= 0.6`：`+1.5`
3. 覆盖率 `>= 0.35`：`0`
4. 低覆盖：`-1.5`
5. 完全无覆盖：`-3`

#### 证据关系一致性

若对象太多但关系太少，则减分；对象少且关系清晰，则加分。

#### 当前局限

这一步虽然用了标签来源记录，但它不是“从标签来源反推链条”，而只是把标签来源当作一个证据密度指标。

### 8.14 时间线裁剪规则

最终写进报告的不是全量支撑事件，而是按优先级裁剪后的时间线。

高优先事件包括：

1. 桥接边上的关键事件
2. 带标签事件
3. 高信号事件类型：
   - `EXEC`
   - `CONNECT`
   - `SEND`
   - `RECV`
   - `WRITE`
   - `CREATE`
   - `READ`
   - `DELETE`
   - `RENAME`
   - `CHMOD`
   - `CHOWN`
4. 高信号对象类别：
   - 临时文件
   - 外部 IP
   - 内部 IP
   - 凭据文件
   - 历史文件
   - 业务文件
   - 持久化文件
   - 提权配置
   - 日志文件

当前默认每条链最多保留：

1. `24` 条时间线事件

## 9. 阶段四：把候选攻击链解释成攻击行为和 ATT&CK 结果

这一阶段对应 `module6_reason`。

### 9.1 为什么不能直接把候选链当最终结论

候选链只说明：

1. 哪些进程、对象、事件可能组成了一条攻击路径

但它还没有回答：

1. 这条路径在攻击语义上代表什么
2. 属于哪个攻击阶段
3. 对应哪些 ATT&CK 战术或技术

因此，还需要把候选链进一步抽象成行为层结论。

### 9.2 当前用了哪些技术

1. 候选链结构化证据打包
2. Holmes 风格行为原子抽取
3. 规则化声明图构造
4. 大模型受约束信息抽取
5. ATT&CK 知识库候选检索
6. 大模型受约束 ATT&CK 映射
7. 行为先验校验与修正

### 9.3 候选链证据包包含什么

系统会把每条候选链整理成一个结构化证据包，主要包括：

1. 路径编号
2. 风险等级和风险分数
3. 阶段覆盖
4. 核心进程列表
5. 桥接边列表
6. 时间线
7. 支撑证据事件
8. 支撑证据对象
9. 支撑证据关系
10. 前导事件
11. 后续动作事件
12. 网络摘要
13. 对象传播摘要
14. 模式标签

#### 动机

这样做是为了让大模型读的是“压缩后的攻击证据单元”，而不是原始日志。

### 9.4 Holmes 风格行为原子是什么

系统会先把候选链抽象成一组固定的中间行为类型，也就是 Holmes 风格行为原子。

这组原子覆盖了：

1. 初始接触
2. 落地执行
3. 建立 foothold
4. 提权
5. 内部侦察
6. 横向移动
7. 任务完成
8. 痕迹清理

#### 这样设计的动机

如果直接从候选链跳到 ATT&CK 技战术，中间缺少一层稳定语义，容易出现：

1. 大模型把同一类证据映射到不同战术
2. 规则层无法解释为什么是这个 ATT&CK

所以中间先引入一层较稳定的“行为原子”。

### 9.5 Holmes 风格行为原子列表

当前版本包括以下行为原子：

1. 接收不可信外部内容
2. 内存变可执行
3. 文件被改成可执行
4. 不可信落地文件被执行
5. 用户打开或执行附件/阶段性对象
6. Shell 或解释器执行命令
7. 外部回连或 C2 通信
8. sudo 提权执行
9. 切换身份提权
10. 读取敏感本地信息
11. 执行侦察命令
12. 网络服务发现/扫描
13. 内网连接/横向
14. 敏感数据外泄
15. 清理日志
16. 删除敏感临时文件
17. 删除可疑落地文件
18. 短生命周期解释器前导链

### 9.6 行为原子匹配规则

系统会先从证据包中抽取一批基础事件集：

1. 外部接收事件集
2. 外部发送事件集
3. 横向连接事件集
4. 可疑执行事件集
5. 敏感读取事件集
6. 日志删除事件集
7. 侦察命令事件集
8. 扫描事件集
9. 内部连接事件集
10. 内存执行迹象事件集
11. `chmod` 事件集
12. 附件迹象事件集
13. 前导事件集
14. 桥接执行事件集
15. 临时文件删除事件集
16. Shell/解释器执行事件集
17. sudo 事件集
18. su/setuid 事件集

然后按以下规则生成行为原子：

1. 有外部接收 -> 接收不可信外部内容
2. 有内存执行迹象，且前面有外部接收或前导事件 -> 内存变可执行
3. 有 `chmod` 且前面有外部接收、桥接执行或前导事件 -> 文件被改成可执行
4. 有桥接执行 -> 不可信落地文件被执行
5. 有附件迹象 -> 用户打开或执行附件
6. 有 Shell/解释器执行迹象 -> Shell 或解释器执行命令
7. 有外部发送，或有外部接收且网络摘要存在 -> 外部回连或 C2 通信
8. 有 sudo -> sudo 提权执行
9. 有 su/setuid -> 切换身份提权
10. 有敏感读取 -> 读取敏感本地信息
11. 有侦察命令 -> 执行侦察命令
12. 扫描事件至少 2 个，或扫描与横向同时存在 -> 网络服务发现/扫描
13. 有内部连接 -> 内网连接/横向
14. 外部发送与敏感读取同时存在 -> 敏感数据外泄
15. 有日志删除 -> 清理日志
16. 临时文件删除且前面有敏感读取 -> 删除敏感临时文件
17. 临时文件删除且前面有桥接执行 -> 删除可疑落地文件
18. 有前导事件 -> 短生命周期解释器前导链

### 9.7 行为原子前置依赖图

为了避免行为原子彼此孤立，系统还建立了前置依赖关系。

例如：

1. “不可信落地文件被执行”依赖于“接收不可信外部内容”或“文件被改成可执行”
2. “Shell 或解释器执行命令”依赖于“不可信落地文件被执行”或“附件执行”或“前导链”
3. “敏感数据外泄”依赖于“敏感读取”和“外部通信”
4. “清理日志”依赖于更早的执行或通信阶段

#### 动机

这样做可以把离散行为原子组织成一条更接近攻击过程的行为图，而不是平铺罗列。

### 9.8 第一次大模型调用：行为确认

这一步并不是让大模型自由总结，而是受严格约束。

#### 大模型输入的内容

1. 候选链证据包
2. 预匹配的 Holmes 风格行为原子

#### 大模型被要求遵守的规则

1. 只能使用证据包中的信息
2. 只能围绕已经给出的行为原子做确认、细化或放弃
3. 不能新造行为类型
4. 不能新造声明编号
5. 不能使用泛化性模糊表述
6. 如果无证据支持，则返回空声明

#### 动机

目的是用大模型做“受约束的语义整理”，而不是“开放式脑补”。

### 9.9 行为声明校验规则

大模型返回后，系统不会直接采纳，而是进行严格校验。

#### 基本合法性校验

1. 必须有声明编号
2. 必须有说明语句
3. 必须引用证据包中的真实事件编号
4. 不能是空泛、模板化陈述

#### 行为类型必要证据校验

不同声明类型还要满足不同必要条件。例如：

1. 接收不可信外部内容 -> 必须看到外部接收
2. 内存变可执行 -> 必须看到 `mprotect`、`virtualalloc` 等语义
3. 文件变可执行 -> 必须看到 `chmod` 或等价语义
4. 不可信文件执行 -> 必须看到可疑执行标签或桥接执行事件
5. Shell/解释器执行 -> 必须看到 `bash`、`python`、`perl` 等语义
6. C2 通信 -> 必须看到外部收发
7. 敏感读取 -> 必须看到凭据/历史/业务数据读取
8. 服务发现/扫描 -> 必须看到横向连接或扫描语义
9. 清理日志/删除文件 -> 必须看到删除、重命名等动作

#### 动机

这一步是为了保证：

1. 大模型只能在真实证据范围内表达
2. 不能把弱证据说成强结论

### 9.10 声明兜底合并规则

系统最终不会只保留“大模型确认结果”，而是做：

1. 规则声明
2. 大模型确认声明
3. 二者合并

如果同一种行为既被规则匹配到，也被大模型确认：

1. 采用更完整的表述
2. 保留更高置信度
3. 合并证据事件编号

#### 动机

规则负责底线可靠性，大模型负责自然语言表达和细节修整。

### 9.11 ATT&CK 候选检索技术

当前版本不会直接让大模型从全部 ATT&CK 中自由选择，而是先做候选检索。

#### 检索输入

系统从证据包和行为声明中抽取：

1. 动作族
2. 行为类型
3. 命令词
4. 对象语义
5. 操作系统提示
6. 行为声明文本

#### 检索方法

系统同时使用：

1. 稀疏检索：TF-IDF
2. 稠密检索：Sentence Transformer 向量

然后再加：

1. 行为兼容性奖励
2. 操作系统一致性修正
3. 行为先验奖励

#### 动机

这样做比单纯字符串匹配更稳，也比让大模型直接从 ATT&CK 全库自由输出更可控。

### 9.12 第二次大模型调用：ATT&CK 映射

在候选 ATT&CK 集合已经检索出来以后，系统再做第二次大模型调用。

#### 大模型被要求遵守的规则

1. 只能使用给定的声明、时间线和 ATT&CK 候选集
2. 每个声明独立映射
3. 先选战术，再选技术
4. 如果技术证据不足，可以只给战术
5. 某些行为有默认偏好的战术方向

#### 典型默认偏好

1. 服务发现、侦察命令 -> 更偏向 Discovery
2. 日志清理、临时文件清理 -> 更偏向 Defense Evasion
3. 附件执行 -> 更偏向 Initial Access + Execution
4. 凭据读取但技术证据弱 -> 宁可只输出 Credential Access 战术

### 9.13 ATT&CK 映射校验规则

系统会对大模型的 ATT&CK 映射再做一层验证。

#### 基本合法性

1. tactic_id 必须满足 `TAxxxx`
2. technique_id 必须满足 `Txxxx`
3. 必须引用有效的行为声明编号

#### 候选集一致性

1. tactic 和 technique 必须来自候选列表

#### 战术技术一致性

1. 若同时给出 tactic 和 technique，则该技术必须支持该战术

#### 行为允许战术约束

每个行为类型都有限制的可映射战术集合。例如：

1. 下载并执行 -> 只允许 Command and Control / Execution
2. 凭据读取 -> 只允许 Credential Access / Collection
3. 业务数据访问 -> 只允许 Collection / Exfiltration
4. 持久化改动 -> 只允许 Persistence / Defense Evasion
5. 日志删除 -> 只允许 Defense Evasion
6. 横向连接 -> 只允许 Lateral Movement

#### 动机

这一步用于防止大模型把语义相近但战术不匹配的技术错误贴上去。

### 9.14 行为先验二次修正规则

如果系统根据行为类型已经有一个非常强的 ATT&CK 先验，而大模型映射结果与之冲突：

1. 冲突映射会被抑制
2. 缺失但高置信的先验映射会被补回

### 9.15 最终输出

每条攻击链最终会输出：

1. 路径摘要
2. 行为声明列表
3. 行为依赖图
4. IOC 列表
5. ATT&CK 候选
6. ATT&CK 最终映射
7. 证据支撑率

## 10. 当前版本实际做过哪些技术对比和取舍

这一节很适合向甲方讲“为什么这样设计”。

### 10.1 全量日志直接分析 vs 任务图驱动分析

#### 全量日志直接分析

1. 优点：理论上信息最全
2. 缺点：噪声极大、成本高、难以解释

#### 当前任务图驱动分析

1. 优点：聚焦高风险区域，效率更高
2. 缺点：依赖上游任务图质量

#### 当前取舍

采用任务图驱动分析，因为它更适合工程落地。

### 10.2 只靠父子进程关系 vs 加入对象传播关系

#### 只靠父子进程关系

1. 优点：简单、稳
2. 缺点：很多真实攻击动作串不起来

#### 当前做法

加入跨对象因果边

#### 当前取舍

通过“同一对象先写后读/执行”构造桥接边，提升攻击链完整性。

### 10.3 直接在原始事件上推理 vs 先做标签和状态压缩

#### 直接在原始事件上推理

1. 优点：不损失细节
2. 缺点：噪声大，难稳定解释

#### 当前做法

先做标准化、去重、对象分类、状态汇总和标签抽象

#### 当前取舍

优先保证结构化和可解释性，再把必要细节通过证据时间线保留下来。

### 10.4 纯规则输出 vs 纯大模型输出

#### 纯规则输出

1. 优点：稳定、可复现
2. 缺点：表达层次有限，难以做细粒度 ATT&CK 映射

#### 纯大模型输出

1. 优点：表达能力强
2. 缺点：不稳定、易幻觉、难验证

#### 当前取舍

采用：

1. 规则抽语义
2. 检索收候选
3. 大模型受约束生成

### 10.5 直接 ATT&CK 映射 vs 先构造行为原子

#### 直接 ATT&CK 映射

1. 优点：链路短
2. 缺点：中间语义层缺失，映射稳定性差

#### 当前做法

先构造 Holmes 风格行为原子，再映射 ATT&CK

#### 当前取舍

这样可解释性更强，也更利于做校验和约束。

从工程角度看，`step2b1` 这一版主要解决了以下问题。

### 11.1 解决了原始日志跨数据集字段不统一的问题

通过标准化事件模型，把不同日志来源统一成同一分析接口。

### 11.3 解决了只看单点异常、无法解释攻击过程的问题

通过候选攻击链，把分散事件拼接成可讲述的因果路径。

### 11.4 解决了只靠父子关系无法串起文件传播的问题

通过跨对象因果边，把对象传播也纳入攻击链。

### 11.5 解决了直接让大模型读原始日志的高噪声问题

通过结构化证据包，让大模型面对的是压缩后的安全语义单元。

### 11.6 解决了 ATT&CK 映射搜索空间过大的问题

通过本地知识库检索先收缩候选，再做大模型映射。

## 12. 当前版本遇到的主要问题和已知限制

这一节建议向甲方坦诚说明，因为这些问题决定了下一步演进方向。

### 12.1 规则文件和真实代码并不完全一致

当前存在一些设计声明和实际实现不完全一致的情况，例如：

1. 某些 YAML 中定义的标签规则没有被完整实现
2. 某些 YAML 中声明的传播规则没有被传播器真正执行
3. 某些评分项在规则表中存在，但代码里没有真正使用

这意味着：

1. 规则文件目前更像“设计说明 + 参数表”
2. 而不是“完全自动执行的规则引擎”

### 12.2 有些标签定义了，但当前主流程几乎不会产出

例如当前版本中，以下标签虽然在规则表里定义了，但代码里没有完整生成逻辑：

1. 可疑控制上下文标签
2. 网络上下文写出文件标签
3. 认证配置对象标签
4. 归档对象标签
5. 安全日志对象标签

#### 影响

1. 这些标签对应的阶段覆盖和评分价值，在当前版本里没有充分发挥

### 12.3 标签来源记录还不是完整的标签因果链

当前标签来源记录主要覆盖：

1. 轻量预标记
2. 进程继承

但没有完整覆盖后续全部行为标签。

#### 影响

1. 目前还不能真正做到“从最终告警标签一路反向回溯到最初触发事件”
2. 标签来源现在更多只是一个辅助证据指标

### 12.4 跨对象因果边虽然有效，但仍然偏保守

当前桥接规则要求：

1. 对象必须是允许对象
2. 时间差较短
3. 语义纪元一致

#### 好处

1. 误报更低

#### 问题

1. 某些攻击如果通过较长时间间隔复用对象，可能会被漏掉

### 12.5 事件聚合和语义去重虽然有效，但可能压掉极细粒度信息

#### 好处

1. 大幅减噪
2. 让链条更可读

#### 风险

1. 某些非常短促但关键的细节，可能在强压缩后被弱化

### 12.6 模式标签和补救规则带有经验性

例如：

1. 前导关键词
2. 行为模式归类
3. 痕迹清理模式

这些规则确实能补链，但也带有场景依赖性。

### 12.7 大模型仍然是最后一层不确定因素

虽然当前做了很多约束：

1. 结构化输入
2. 候选集约束
3. 声明校验
4. 映射校验

但大模型输出依然不可能像纯规则那样完全确定。

#### 当前应对方式

1. 先规则抽原子
2. 再候选检索
3. 再大模型受约束映射
4. 最后再做规则校验

## 13. 与传统方案相比，这个版本的价值在哪里

如果要向甲方强调“项目价值”，可以从以下几点来讲。

### 13.1 不只是“发现异常”，而是能输出攻击过程

很多系统只给出：

1. 某个分数高
2. 某个节点可疑

但当前系统能输出：

1. 哪些进程参与了攻击
2. 哪些对象承载了传播
3. 哪些事件构成了证据
4. 这些证据对应哪些攻击行为

### 13.2 不是黑箱系统，解释链完整

系统的中间产物非常丰富：

1. 标准化事件
2. 进程状态
3. 对象状态
4. 标签
5. 对象版本
6. 候选攻击链
7. 行为声明图
8. ATT&CK 映射

这意味着：

1. 结果可复盘
2. 规则可调
3. 错误可定位

### 13.3 兼顾结构化安全逻辑和语义理解能力

规则层负责：

1. 稳定性
2. 安全约束

大模型层负责：

1. 高层行为解释
2. 攻击语义整合
3. 更自然的输出表达

### 13.4 对复杂攻击比单点告警更友好

很多复杂攻击不是一个点，而是一段过程：

1. 外部接触
2. 落地执行
3. 本地收集
4. 横向或外传
5. 痕迹清理

当前系统正是围绕“过程”来建模的，而不是围绕“单点异常”来建模。
